Recovery Logic — 6-Stage Protocol
Per-scenario recovery playbook · 6-stage sequence · automated with manual escalation for severe cases
RECOVERY STAGES 1-3
Detect · Halt · Flatten
DETECT
Stress condition detected by automated monitor (drawdown, spread, latency, connection, vol, gap). Monitor publishes STRESS_DETECTED event to NATS.
Latency: < 100 ms from condition onset
HALT
Halt new entries immediately. Atomic flag set in ExecutionEngine. No new orders accepted. Existing orders remain open.
Latency: < 50 ms from detect
FLATTEN
Flatten all open positions via market orders if MDD > 5% or kill-switch criteria met. Otherwise, hold positions with tightened stops.
Latency: < 500 ms from halt (kill-switch SLA)
RECOVERY STAGES 4-6
Protect · Recover · Resume
PROTECT
Activate protective mode: reduced position size (50%), widened stops (1.5×), no new entries. Mode persists for 24 h or until cleared by operator.
Duration: 24 h minimum
RECOVER
Verify stress condition cleared: spread < 2× baseline, latency < 200 ms, broker reconnected, vol < 2× avg, gap absorbed. Recovery check runs every 30 s.
Duration: until all clear for 60 s
RESUME
Resume normal trading with 50% size for first hour, 75% for next 4 h, 100% after 24 h if no further stress. Operator notified.
Staircase: 50% → 75% → 100%
RECOVERY INVARIANTS · NEVER VIOLATED
5 invariants hold throughout the recovery protocol
① Kill-switch SLA: flatten completes < 500 ms from trigger (measured by dedicated latency probe)
② Position reconciliation: post-flatten, local position state == broker position state (zero phantom orders)
③ Audit trail: every recovery action logged with timestamp, reason, and operator ID (if manual)
④ No silent resume: resumption requires explicit "all clear" verification, never a timer
⑤ Operator notification: P2 PagerDuty alert at DETECT, P1 if FLATTEN triggers, P1 if not recovered in 30 min
